# Requirements Document

## Introduction

This feature covers three related improvements to the GrapesJS email builder:

1. **Professional Test Email Template** — A ready-to-use default template that loads via the Templates modal, built entirely from existing builder components (header, heading, paragraph, image, button, social icons, footer text). It gives users a clean, modern, responsive starting point for real-world email testing.

2. **Email-Safe Social Icons** — The current social component renders icons as inline SVGs via `getSocialIconSvg()` inside `renderSocialLinks()` in `minimal-builder-plugin.js`. Inline SVG is not supported in Gmail, Outlook, Apple Mail, or Yahoo Mail, so icons are invisible in received emails. This fix replaces inline SVGs with email-safe alternatives (publicly hosted PNG images or Unicode/emoji fallbacks with text labels) so icons render correctly across all major email clients.

3. **Save Template Feature** — A "Save Template" button in the top bar (appbar actions area in `shell.js`) that lets users name and persist the current editor state to localStorage. Saved templates appear in the existing Templates modal alongside built-in templates and can be loaded or deleted.

## Glossary

- **Builder**: The GrapesJS-based email editor application defined across `app.js`, `shell.js`, `config.js`, and `minimal-builder-plugin.js`.
- **Template**: A named email design consisting of GrapesJS project data (JSON), production HTML, and CSS, stored either as a built-in definition in `emailTemplates` or as a user-saved entry in localStorage.
- **Built-in Template**: A template defined in the `emailTemplates` object inside `minimal-builder-plugin.js` and exposed via `minimalBuilder.emailTemplates`.
- **Saved Template**: A user-created template persisted to localStorage, keyed by a user-supplied name and a creation timestamp.
- **Templates Modal**: The existing `#template-modal` dialog in `shell.js` that lists available templates and allows the user to load one into the editor.
- **Production HTML**: The fully inlined, email-client-safe HTML string produced by `buildProductionHtml()` in `app.js`.
- **GrapesJS Project Data**: The serialised editor state returned by `editor.getProjectData()` and restored by `editor.loadProjectData()`.
- **Social Component**: The GrapesJS component of type `email-social` whose export HTML is generated by `renderSocialLinks()` in `minimal-builder-plugin.js`.
- **renderSocialLinks**: The function in `minimal-builder-plugin.js` that produces the table-based HTML for the social icons row sent in emails.
- **getSocialIconSvg**: The helper in `minimal-builder-plugin.js` that currently returns an inline SVG string for a given social network — the root cause of invisible icons in email clients.
- **Email-Safe Icon**: A social network icon rendered as a publicly hosted PNG/GIF `<img>` tag, or as a Unicode/emoji character with a visible text label, both of which are supported by Gmail, Outlook, Apple Mail, and Yahoo Mail.
- **Top Bar / Appbar**: The `<header class="studio-appbar">` element in `shell.js` that contains the brand mark, Templates button, save status indicator, Send Test Email button, and EXPORT button.
- **localStorage**: The browser's `window.localStorage` API used for persisting editor state between sessions.
- **Save Template Modal**: A new modal dialog (or inline prompt) triggered by the "Save Template" button that collects a template name from the user before saving.

## Requirements

### Requirement 1: Professional Test Email Template

**User Story:** As an email designer, I want a ready-to-use professional template available in the Templates modal, so that I can start from a polished, real-world layout without building one from scratch.

#### Acceptance Criteria

1. THE Builder SHALL include a built-in template named "Professional" (or equivalent) in the `emailTemplates` object inside `minimal-builder-plugin.js`.
2. WHEN the Templates modal is opened, THE Templates_Modal SHALL display the Professional template as a selectable card alongside existing built-in templates.
3. WHEN the user selects the Professional template, THE Builder SHALL load it into the editor canvas using the same `applyTemplate` flow used for all other built-in templates.
4. THE Professional_Template SHALL be composed exclusively of existing GrapesJS component types already registered by `minimal-builder-plugin.js` (such as `site-header`, `email-heading`, `email-paragraph`, `email-image`, `email-button`, `email-social`, and `site-footer`), with no new component types introduced.
5. THE Professional_Template SHALL include at minimum: a header/brand area, a heading block, a paragraph/body copy block, an image block, a button (CTA) block, a social icons block, and a footer text block.
6. WHEN the Professional template is exported via `buildProductionHtml()`, THE Builder SHALL produce valid, table-based HTML with no inline SVG elements in the output.
7. THE Professional_Template SHALL apply consistent padding and spacing values so that the rendered email has a clean, modern appearance at 600 px content width.
8. WHEN the Professional template is previewed on the mobile device breakpoint (360 px), THE Builder SHALL render the template in a single-column stacked layout without horizontal overflow.

### Requirement 2: Email-Safe Social Icons

**User Story:** As an email sender, I want social icons in the social component to render visibly in Gmail, Outlook, Apple Mail, and Yahoo Mail, so that recipients can see and click the social links in received emails.

#### Acceptance Criteria

1. WHEN `renderSocialLinks()` generates HTML for the social component, THE Social_Component SHALL NOT use inline SVG markup (`<svg>` elements) in the produced HTML string.
2. WHEN `renderSocialLinks()` generates HTML for a social item, THE Social_Component SHALL render each icon as either:
   a. An `<img>` tag referencing a publicly hosted PNG or GIF URL (minimum 32 × 32 px), OR
   b. A Unicode character or emoji with a visible text label as a fallback.
3. WHEN an `<img>`-based icon is used, THE Social_Component SHALL include a descriptive `alt` attribute on each `<img>` tag containing the social network name (e.g., `alt="Facebook"`).
4. WHEN an `<img>`-based icon is used and the image fails to load, THE Social_Component SHALL display the social network name as visible text via the `alt` attribute so the link remains usable.
5. THE Social_Component SHALL preserve all existing configurable properties: `iconSize`, `iconStyle`, `iconShape`, `spacing`, `align`, and `linkColor`.
6. WHEN `iconSize` is set to a value between 16 and 64 (inclusive), THE Social_Component SHALL render each icon at the specified pixel dimensions.
7. THE Social_Component SHALL support at minimum the following networks with email-safe icons: Facebook, Instagram, LinkedIn, YouTube, X (Twitter), TikTok, Pinterest, and Snapchat.
8. WHEN the social component HTML is sent via the test email flow, THE Social_Component SHALL render visible icons in Gmail, Outlook (2016+), Apple Mail, and Yahoo Mail without requiring any client-side JavaScript.
9. THE Social_Component SHALL continue to render correctly in the GrapesJS canvas preview (editor view) after the SVG replacement.

### Requirement 3: Save Template Feature

**User Story:** As an email designer, I want to save my current editor state as a named template, so that I can return to it later or reuse it as a starting point for future emails.

#### Acceptance Criteria

1. THE Builder SHALL display a "Save Template" button in the appbar actions area of `shell.js`, positioned between the "Send Test Email" button and the "EXPORT" button.
2. WHEN the user clicks "Save Template", THE Save_Template_Modal SHALL open and present an input field for the user to enter a template name.
3. WHEN the user submits a non-empty template name, THE Builder SHALL save the following data to localStorage under a key that includes the template name and a creation timestamp:
   a. The GrapesJS project data returned by `editor.getProjectData()`,
   b. The production HTML string returned by `buildProductionHtml()`,
   c. The current CSS string from the editor.
4. WHEN the user submits an empty template name, THE Save_Template_Modal SHALL display a validation error message and SHALL NOT save any data.
5. WHEN a template is saved successfully, THE Builder SHALL close the Save_Template_Modal and display a toast notification confirming the save.
6. WHEN the Templates modal is opened, THE Templates_Modal SHALL display all saved templates in a visually distinct section (e.g., labelled "My Templates") separate from the built-in templates section.
7. WHEN the user clicks a saved template card in the Templates modal, THE Builder SHALL load the saved GrapesJS project data into the editor via `editor.loadProjectData()` and close the modal.
8. WHEN the user loads a saved template, THE Builder SHALL restore the body settings (text color, background, content width, font, etc.) to the values stored with that template.
9. WHEN a saved template card is displayed, THE Templates_Modal SHALL show a delete control (e.g., a "×" button) on each saved template card.
10. WHEN the user activates the delete control on a saved template, THE Builder SHALL remove that template's entry from localStorage and re-render the Templates modal without the deleted template.
11. THE Builder SHALL store saved templates under a localStorage key namespace that does not conflict with the existing editor autosave key (`grapesjs-email-builder-studio-v*`).
12. WHEN localStorage is unavailable or throws an error during save, THE Builder SHALL display an error toast message and SHALL NOT crash.
13. WHEN localStorage is unavailable or throws an error during load, THE Builder SHALL display an error toast message and SHALL NOT crash.
14. WHEN the Templates modal is opened and no saved templates exist, THE Templates_Modal SHALL NOT render the "My Templates" section heading or any empty placeholder cards for saved templates.
